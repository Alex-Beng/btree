A simple set

The library provides associative containers that are resident on some form of random access external memory, such as a hard-disk, solid-state disk, or USB memory-stick.  Here is an example, set1.cpp:

#include <boost/btree/btree_set.hpp>
#include <iostream>

using std::cout;
using namespace boost::btree;

int main()
{
  typedef btree_set<int> BT; 
  BT bt("int_set.btr", flags::truncate);

  bt.insert(5);
  bt.insert(3);
  bt.insert(1);

  for (BT::iterator itr = bt.begin(); itr != bt.end(); ++itr)
    cout << *itr << '\n';

  cout << "lower_bound(3) is " << *bt.lower_bound(3) << '\n';
  cout << "upper_bound(3) is " << *bt.upper_bound(3) << '\n';
}

This should be familiar to users of the C++ standard library associative container std::set. The only real difference, other than header and class names, is the constructor call. Because the container is disk-resident, an argument supplies a path for the file to be created. A second argument supplies flags that specify details for opening the file.

Build and test set1.cpp like this (Windows):

>cd boost-root/libs/btree/example   Works for both Windows and POSIX
>b2 set1 install
... build output elided ...
>bin\set1                           On POSIX: bin/set1
1
3
5
lower_bound(3) is 3
upper_bound(3) is 5

Go ahead and try this yourself! Once you have built and tested set1.cpp as is, try experimenting with some of the other functions already familiar to you from std::set. Did they work as you expected? Hopefully your answer was "yes"!

What is the point of using a btree_set instead of a std::set? Let's look at another example...

Reading from a B-tree file

Here is the sample code for set_read_file.cpp, showing edits from set1.cpp:

#include <boost/btree/btree_set.hpp>
#include <iostream>
#include <boost/detail/lightweight_main.hpp> 

using std::cout;
using namespace boost::btree;

int cpp_main(int, char *[])
{
  typedef btree_set<int> BT;
  BT  bt("int_set.btr");

  for (BT::iterator itr = bt.begin(); itr != bt.end(); ++itr)
    cout << *itr << '\n';

  cout << "lower_bound(3) is " << *bt.lower_bound(3) << '\n';
  cout << "upper_bound(3) is " << *bt.upper_bound(3) << '\n';
}

Since the "set1.btr" file already exists, we let the constructor flags default to flags::read_only and don't redo the inserts.

That brings us back to the original point; B-tree are disk resident, 
  and that makes them very different animals from standard library associative 
  containers.

  
  
What about the additional template parameters, constructor arguments, 
  and member functions that are not present in std::set? You may have spotted 
  these if you looked at the
  reference documentation. Don't worry about them 
  for now; the defaults will get you started, and most of the unfamiliar 
  parameters, arguments, and functions are used only when 
  tuning for ultimate performance.

  
  
What you probably do want to learn about right away is how to cope with the 
  very stringent requirements the library places on template parameter Key (and 
  T, for maps), since these requirements will cause endless problems if you 
  don't understand them. Read on...

  
  
String set first try

  
  
An obvious transformation of set1.cpp to handle std::string instead of
  int 
  looks like example set2_1st_try.cpp:

  
  

    
#include <boost/btree/btree_set.hpp>
#include <string>
#include <iostream>
#include <boost/detail/lightweight_main.hpp> 

using std::cout;
using namespace boost::btree;

int cpp_main(int, char *[])
{
  typedef btree_set<std::string> set_type;
  set_type set("string_set.btr", flags::truncate);

  set.insert("eat");
  set.insert("drink");
  set.insert("be merry");

  for (set_type::iterator it = set.begin(); it != set.end(); ++it)
    cout << *it << '\n';
}


  

  
  
But this code will not compile! The error message from Visual C++ 2012 
  looks like this:

  
  

  
  
..\..\..\boost/btree/btree_set.hpp(120) : error C2338: btree Key must not be std::string
set2_1st_try.cpp(16) : see reference to class template instantiation 'boost::btree
::btree_set<Key>' being compiled
with
[
Key=std::string
]
...

  

  
The problem is that std::string 
  does not conform to the
  requirement 
  that Key be fixed-length, 
  memcpyable, and not container pointers or references except in a proxy class. How can those 
  requirements be met? Read on...

  
  
Small strings

  
  
Many applications involve only small strings; some std::basic_string 
  implementations acknowledge this by employing a small-string optimization 
  under the covers. The class template 
  string_box provides fixed length strings, with the maximum 
  length supplied by a template parameter.

  
  

    
#include <boost/btree/btree_set.hpp>
#include <boost/btree/support/string_box.hpp>
#include <iostream>
#include <boost/detail/lightweight_main.hpp> 

using std::cout;
using namespace boost::btree;

int cpp_main(int, char *[])
{
  typedef btree_set<string_box<16> > set_type;
  set_type set("string_set.btr", flags::truncate);

  set.insert("eat");
  set.insert("drink");
  set.insert("be merry");

  for (set_type::iterator it = set.begin(); it != set.end(); ++it)
    cout << *it << '\n';
}

  

  
  
  
Here is the output:

  
  

    
be merry
drink
eat

  

  
  
Longer strings

  
  
Except when the elements are fixed-length and quite short relative to disk 
  page size, the best use of a B-tree is as an index into a random access file. 
  That was described in the introduction of Bayer's original paper, 
  Organization and Maintenance of Large Ordered Indexes.

  
  
This is implemented as two files, a btree_set (or btree_multiset) 
  file used as an index pointing into a random-access flat file which contains 
  the actual data. Databases and other software often use a similar approach, 
  but may hide the index file from end-user view. The benefits of this two file 
  approach include:

  
  
    
    
  • 
  
    The elements in the flat file can be variable length.
    • 
  
  • 
  
    The elements in the flat file can be far larger than would be 
  practical for elements in a B-tree file.
    • 
  
  • 
  
    Multiple B-tree index files can exist for a single flat file.
    • 

  • 
  
    Off-the-shelf or custom-written software can also access the flat 
  file.
    • 


  
This two layer approach uses some additional components to create 
  a seamless interface:

  
  
    
    
  • A Boost implementation of the new standard library 
    
    string_view class to act as the container's 
    Key type, so that the familiar
    std::string operations 
    are available to users.
    • 
    
  • An adapter index traits class to copy data into and out of the 
    flat file.
    • 
    
  • Class btree_index_set to glue the pieces together into a 
    seamless interface.
    • 


  
Here is the final version of set2.cpp:

  
  

    
#include <boost/btree/btree_index_set.hpp>
#include <boost/btree/support/string_view.hpp>
#include <iostream>
#include <boost/detail/lightweight_main.hpp>

using std::cout;
using namespace boost::btree;

int cpp_main(int, char *[])
{
  typedef btree_index_set<string_view> BT;
  BT bt("string_index_set", flags::truncate);

  set.insert("eat");
  set.insert("drink");
  set.insert("be merry");

  for (BT::iterator it = set.begin(); it != set.end(); ++it)
    cout << *it << '\n';
} 

  

  
  
  
Here is the output:

  
  

    
be merry
drink
eat


  

  
....And a dump of the flat file:

  

    
00000000  0365 6174 0564 7269 6e6b 0862 6520 6d65 .eat.drink.be me
00000010  7272 79                                 rry

  

  
Notice that each element begins with a count. That's more efficient than 
  null termination when processing long strings, and allows strings to contain 
  embedded nulls. The count is itself variable length, and will use however many 
  count bytes is required by the size of the string. The 0x80 bit will be set 
  for count bytes except the last (i.e. low-order).
